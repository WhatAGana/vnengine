**얇게 간다** — 소문수집·마을파견 기믹은 별도(07-D2/나중). C가 기다리는 건 "인과율 수급 코어"뿐.

## 프롬프트 본문

```
06 + 07-A1 + 07-A2 + 07-B가 main에 있는 상태에서, 07-D "여관 인과율 엔진(수급 코어)"을
subagent-driven-development로 완주해줘.

기존 원칙 필수: ① 순수 코어(UnityEngine·System.IO 미참조) ② 불변 상태 ③ 데이터 주도(계수·상한·곡선을 데이터로).

스코프: 여관 상태(직원/내구도/메뉴) + 웨이브당 손님 → 인과율/골드 수급 산식.
소문수집·마을파견·모험가예보 기믹은 이 슬라이스 밖(별도). 인과율 "소비"(스탯강화/레벨업)와 골드 환산 배선은 07-C.
여기선 "여관이 회차 진행 중 인과율과 골드를 얼마나 생산하나"까지.

## 설계 결정 (확정 — brainstorming 불필요, plan→구현)

### 여관 상태 (메타 귀속 — 회차 넘어 유지)
InnState { int Staff; int Decor; int MenuLevel; }  // 06 MetaState에 추가
- 06 런/메타 분류상 여관은 회차 넘어 유지 = 메타. (직원 고용·메뉴 개발은 영속 투자)
- 불변: 변경은 새 InnState 반환. 방어적 복사(06 원칙).

### 여관 수급 산식 (완화+상한 — 시뮬 6차 검증, 인플레 방지 핵심)
웨이브(또는 정비일)당:
  if Decor > 0:
    guests    = min(Staff*2 + MenuLevel, 25)          // 손님수: 여관규모에만 연동(던전레벨 무관!), 상한 25
    innGold   = min( isqrt(guests)*8 + MenuLevel*3, 300 )   // 골드: 제곱근+상한 (여관은 돈벌이 주력 아님)
    innKarma  = guests                                 // 인과율: 손님수 그대로 (레벨/스탯강화 주력 축)
    Decor    -= (자연감소)                              // 내구도 감소
  else:
    (손님 안받음 — 수입 0. 내구도 게이트)
- ★ 손님수를 던전레벨과 분리(시뮬 5차 폭주 교훈: 던전레벨 연동하면 후반 여관수입 폭증).
- ★ 새 수입원엔 기존 수입원과 같은 완화/상한(제곱근+상한): 여관 골드비중 목표 <10%.
- 계수(2,25,8,3,300 등)는 전부 데이터/튜닝대상. isqrt는 A1의 정수제곱근 재사용.

### 여관 성격 = 인과율 엔진 (세계관·시스템 정합)
- 여관의 주 산출은 골드가 아니라 인과율(innKarma). 돈은 전투(약탈)에서, 성장(인과율)은 여관에서 = 역할분리.
- 세계관: 여관 = 코어 인과율 배출구. 손님 만족 = 인과율 배출 = 던전 안정.

### 내구도 게이트
- Decor가 0이면 손님 안 받음(수입 0). 정비 시 수리(골드 소모는 07-C, 여기선 "Decor>0이어야 수급" 게이트 구조만).
- Decor 자연감소는 웨이브/일 단위 데이터.

## 작업 분해 (subagent-driven)
1. InnState.cs (불변, 메타 귀속) + 06 MetaState 배선 + 세이브(평면 직렬화)
2. InnIncomeRule.cs (순수함수): (InnState) → (innGold, innKarma). 완화+상한 산식, 데이터 계수
3. 내구도 게이트 + 자연감소 규칙
4. 문서(07 §7 갱신)

## 검증 (EditMode)
  - InnState 불변 + 세이브 왕복 일치(방어적복사)
  - 손님수가 던전레벨 올려도 안 변함(여관규모에만 연동) ← 시뮬5차 폭주 방지 핵심
  - innGold 상한 300 작동(던전레벨 올려도 여관골드 폭증 안함)
  - Decor=0이면 수입 0 (게이트)
  - guests 상한 25
  - innKarma = guests (인과율이 주 산출)
  - 정수결정론·불변·Core순수성
