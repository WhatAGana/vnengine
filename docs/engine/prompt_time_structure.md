# 클로드 코드 인계 프롬프트 — 1회차 시간구조 (90일 루프 + 모듈형 진행 컨트롤)

06~07-C(게임 루프 로직) 위에 얹는 **1회차 뼈대의 첫 조각**. 지금까지 조각들을 90일 시간구조로 묶는다.
참조: story_bible v2.14 §6.1(1회차), story_system_mapping.md §3(1회차 필요 목록).

## 프롬프트 본문

```
06 + 07-A1/A2/B/C가 main에 있는 상태에서, "1회차 시간구조(90일 루프 + 모듈형 진행)"을
subagent-driven-development로 완주해줘.

기존 원칙 필수: ① 순수 코어(UnityEngine·System.IO 미참조) ② 불변 상태 ③ 데이터 주도.

## 핵심 설계 원칙: 시간 진행(로직)과 소비 방식(속도/스킵)을 분리
- 코어는 "하루 전이" 하나. 속도·스킵을 절대 코어에 박지 말 것.
- 빠른재생·구간스킵은 새 로직이 아니라 "AdvanceDay를 어떤 리듬으로/언제까지 호출하느냐"일 뿐.
- → 코어 무변경으로 새 진행모드를 컨트롤러에 추가할 수 있는 모듈 구조.

## 90일 구조 (story §6.1)
- 90일 = 9주기 × 10일.
- 각 주기: 1~9일 정비 페이즈 / 10일 웨이브(제국군 확정). 정비일엔 모험가 웨이브 확률 난입 가능.
- 웨이브 날 = 정비 불가. 9일차에 세이브 확인.
- 90일 종료(또는 코어 자연붕괴) → 회귀(06 Regress).

## 작업 1 — 시간 코어 (순수, Core/Sim/Time/**)
TimeState { int Day; }  // 06 RunState에 편입(회차 리셋). Day 1~90.
AdvanceDay(campaign) → campaign  // 순수함수:
  - Day += 1
  - 그날 페이즈 판정(정비/웨이브)에 따라 처리:
    · 웨이브날(10·20…90일): CampaignWaveRule.ResolveWave 호출(전투·약탈·포획·여관수급)
    · 정비날: 여관 수급(InnIncomeRule)·내구도 감소 등 정비 페이즈 자원 처리
      ⚠ 07-D 주의: 여관 게이트 판정(Decor>0)을 Decay 前에.
  - Day > 90 (또는 코어붕괴 조건) → 회귀 트리거 신호 반환(실제 Regress는 caller가)
  ※ AdvanceDay는 속도·스킵을 전혀 모른다. 하루 전이 + 그날 사건 처리까지만.

## 작업 2 — 순수 질의 함수 (스킵/자동이 참조)
Core/Sim/Time/TimeQuery:
  GetPhase(day) → Maintenance | Wave
  IsWaveDay(day) → bool
  DaysUntilNextWave(day) → int
  IsSaveDay(day) → bool  // 9일차(각 주기 9·19…)
  전부 순수·결정론.

## 작업 3 — 진행 컨트롤러 (모듈, AdvanceDay 조합)
Core/Sim/Time/TimeController (또는 Unity 레이어 — 순수성 유지되게):
  Step()              // 하루: AdvanceDay 1회
  SkipToNextWave()    // 다음 웨이브 전날까지 AdvanceDay 루프 (웨이브날엔 반드시 정지)
  SkipToDay(n)        // 특정 일차까지 (n이 웨이브날 지나면 그 웨이브날에 정지)
  // FastForward(속도)는 UI 틱 레이어 — 로직은 Step 반복, 속도는 표시 계층에서(코어 아님)
  ★ 스킵 = 정산 방식: 건너뛴 날의 AdvanceDay를 실제로 다 호출(여관수급·내구도·자원이 실제 누적).
    화면만 안 보여주고 계산은 다 함. "모르는 새 관리 회피" 방지.
  ★ 웨이브 날엔 자동 정지: 스킵은 정비 구간만. 웨이브 만나면 멈춤(story "웨이브날 정비불가"와 정합).
  ★ 새 진행모드(예: SkipToEvent)는 이 컨트롤러에 메서드 추가로 끝. 코어(AdvanceDay) 무변경.

## 작업 4 — 회귀 연동
- Day>90(또는 붕괴 조건) → 06 Regress 호출로 1회차→다음 회차. TimeState.Day도 1로 리셋(런 귀속).
- 회귀 후 90일 다시 시작.

## 작업 5 — 대시보드 연동 + 문서
- 기존 검증 대시보드에 진행 컨트롤 버튼 추가: "하루" "다음 웨이브까지" "빠른재생 토글".
  현재 Day/페이즈(정비/웨이브)/다음웨이브까지 표시. (대시보드는 검증용 — 가볍게)
- 07 문서에 시간구조 절 추가.

## 검증 (EditMode)
  - AdvanceDay가 Day 정확히 +1, 90 넘으면 회귀신호
  - 웨이브날(10·20…)에 ResolveWave 호출, 정비날엔 안 함
  - 정비날 여관수급/내구도 처리, 게이트가 Decay 前
  - SkipToNextWave가 웨이브 전날 정지(웨이브날 안 넘김)
  - 스킵 정산: 건너뛴 날 자원변화가 실제 누적(하루씩 진행과 동일 결과) ← 정산방식 핵심
  - SkipToDay(n)이 n 전 웨이브날에 정지
  - 회귀 후 Day=1, 런 리셋/메타 유지
  - AdvanceDay 순수·불변·결정론(같은시드 같은결과)
  - 코어(AdvanceDay)에 속도/스킵 개념 없음(구조 검증)

## 진행: subagent-driven(plan→구현→sonnet→검증→opus브랜치→전건통과→main --no-ff). 무관 워킹트리 미커밋.

## 설계 노트 (당신용)

- **모듈형의 핵심 = AdvanceDay 하나만 코어, 나머지는 조합**: 빠른재생·스킵을 코어에 박으면 새 모드마다
  코어 수정. AdvanceDay를 "하루 전이" 순수함수로 두면 모든 모드가 그 위 조합 → 코어 무변경으로 확장.
- **스킵 = 정산 방식**: 건너뛴 날 실제 계산(여관·내구도 흐름). 안 그러면 스킵으로 관리 회피 가능.
  웨이브날 자동정지로 "웨이브날 정비불가"(story §6.1) 지킴.
- **FastForward는 코어 아님**: 빠른재생은 로직상 Step 반복일 뿐, "빠름"은 UI 틱 속도(표시 계층). 코어는
  속도를 몰라야 함. 이걸 섞으면 결정론·테스트가 깨짐.
- **이게 1회차 뼈대의 유일한 신규 작업**: 나머지(전투·여관·포획·가챠·회귀)는 07까지 다 있음. 시간구조가
  그것들을 90일로 묶는 마지막 조각. 이거 끝나면 "서사 씬(.vns)"만 얹으면 1회차 완주.
- **다음**: §6.1 서사 씬(.vns) — 기상→편지→미갈→튜토소환("무언가의 의지")→회귀. 시간구조의 특정 시점에 훅.
```
