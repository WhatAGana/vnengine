# 클로드 코드 인계 프롬프트 — 07-B: 배치 예산제 & 방 그래프 & 포획 정식화

06 + 07-A1(8스탯) + 07-A2(전투코어) 위에 얹는 슬라이스. A2가 "선형 최소 가정"으로 짠 방을
정식 그래프 구조로 승격하고, 배치 예산제와 **포획 조건 정식화**(→ 07-C 인과율 수급의 실체)를 만든다.

---

## 프롬프트 본문

```
06 + 07-A1 + 07-A2가 main에 있는 상태에서, 07-B "배치 예산제 & 방 그래프 & 포획 정식화"를
subagent-driven-development로 완주해줘.

기존 원칙 필수: ① 순수 코어(UnityEngine·System.IO 미참조) ② 불변 상태 ③ 데이터 주도(코스트·상한·조건을 데이터로).

스코프: 방 그래프 구조 + 몹 배치 예산제 + 던전레벨=방상한 + 포획 조건 정식화.
전투 규칙 자체는 A2에 있음(여기선 그 위에 "어디에 뭘 배치하나"와 "무엇을 포획하나"를 얹음).
골드/인과율 환산·레벨업은 07-C, 여관은 07-D.

## 설계 결정 (확정됨 — brainstorming 불필요, 곧장 plan→구현)

### 방 구조: 그래프 자료구조 + 1편 선형 배치
- 방을 **그래프 노드**로 정식화: RoomNode { RoomId Id; MonsterSlots; TrapSlots; NextRooms[] }
- A2가 "선형 최소 가정"으로 순회하던 걸, 이 그래프가 대체(선형은 NextRooms가 1개인 특수케이스).
- **1편 기본 배치는 선형**(입구→방1→…→코어). 단 자료구조는 분기·병렬 경로 확장 가능하게 열어둠.
  → 8스탯과 같은 원칙: 골격은 확장 가능, 1편 노출은 단순. 확장판 그물망 구조를 나중에 데이터로.
- A2 CombatResolver가 이 그래프를 순회하도록 연결(선형가정 → 그래프순회로 승격).

### 배치 예산제 (포인트 바이)
- **총 예산 = 현재 열려있는 방 개수 × 3** (최대방 아님 — 현재방 기준. 최대기준이면 "방 최소, 몹 최대"가 정답이 돼 방시스템이 죽음)
- 각 몹은 **코스트(레어도)**: MonsterDef에 Cost 필드. 예 임프1/고블린1/오크2/하이오크3/서큐버스3/데스나이트4/고위마족5 (데이터)
- 예산 내 자유 배치: 배치 시 코스트 합이 예산 초과 불가. 한 방에 몰거나 분산하거나 자유.
- 주인공은 **예산 제외**, 단 **코어룸 1칸 앞 방까지만** 배치 가능(최종 방어선). — A2에서 주인공 요격은 이미 있음, 여기선 배치 위치 제약만.

### 방 확장 (던전레벨 = 상한)
- rooms_cap = 3 + dungeonLevel*2 (lv1=5, lv2=7… 데이터 곡선)
- **상한과 실제 건설 분리**: 던전레벨=지을 수 있는 천장, 실제 건설=그 안에서 골드 소모(골드는 07-C, 여기선 "천장 안에서 방 개수 관리" 구조만).
- 코어룸(마지막 노드) 피격 = 회차 실패 신호(A2 CoreHit 재사용).

### ★ 포획 조건 정식화 (B의 핵심 산출물 — C의 인과율 수급이 여기 의존)
A2에 포획분기(CanBeCaptured + 조건 → Captured)가 이미 있음. B에서 "조건"을 정식 규칙으로 확정:
- CaptureRule: 침입자를 처치 대신 포획하는 조건을 데이터+규칙으로 정의.
  1편 포획 트리거 예시(데이터로): 함정으로 마지막 타격 / 특정 포획능력 몹(서큐버스류)이 처리 / 주인공이 직접 제압(코어앞).
- 네임드 플래그: 침입자 생성 시 IsNamed 플래그(잡졸/네임드 구분). 네임드는 포획 시 별도 처리 경로(히로인 이벤트 훅), 잡졸은 감옥 인원(경제)으로.
  → 이 구분이 07-C(인과율은 방면량, 히로인은 네임드만 트리거)의 전제.
- CaptureResult가 RunState.Captives에 쌓이도록(06 상태). 네임드/잡졸 구분 필드 포함.
- ※ 네임드 "회차 리셋 여부"는 서사 미확정 → 플래그만 열어두고 로직 강제하지 말 것(생성 시 IsNamed + ResetPolicy 필드 자리만).

## 작업 분해 (subagent-driven)
1. RoomNode/RoomGraph (그래프 구조, 선형은 특수케이스) + A2 CombatResolver를 그래프순회로 연결
2. MonsterDef.Cost + PlacementBudget (현재방×3, 코스트합 검증, 주인공 코어앞1칸 제약)
3. DungeonLevel→rooms_cap 상한 + 방 개수 관리(건설은 천장 안에서, 골드소모는 C)
4. CaptureRule 정식화 + 네임드/잡졸 구분 + RunState.Captives 배선
5. 문서(07 §4·§13·§14 갱신)

## 검증 (EditMode)
  - 그래프 순회가 A2 선형결과와 동일(회귀 없음) + 분기 노드에서도 순회 정확
  - 배치 코스트합 > 예산이면 배치 거부
  - 예산 = 현재방개수×3 정확 (방 늘리면 예산 증가)
  - 주인공 코어앞1칸 외 배치 거부
  - rooms_cap이 던전레벨 상한 정확
  - 포획조건 충족 시 Captured, 미충족 시 Killed
  - 네임드/잡졸 플래그가 Captives에 정확히 구분되어 쌓임
  - 결정론·불변·Core순수성 유지

## 설계 노트 (당신용)

- **왜 그래프 자료구조인데 선형 배치인가**: story_bible 스크린샷의 그물망은 확장판/후반 콘텐츠. 1편은 선형이면
  충분한데, 자료구조를 선형으로 짜버리면 나중에 그래프로 못 늘려 재작업. 8스탯과 동일 원칙으로 골격만 열어둠.
- **포획 정식화가 B의 진짜 산출물**: 클로드 코드도 짚었듯 C의 인과율 수급(포로 방면)이 여기 의존. B에서 "무엇이
  포획되고, 네임드/잡졸을 어떻게 구분하나"를 확정해야 C가 실체를 가짐. 그래서 CaptureRule에 무게.
- **네임드 회차 리셋은 열어만 둠**: story_bible 미확정 사항. 코드가 강제로 정하지 않게 플래그 자리만. 서사 확정
  후 세팅(지금까지 클로드 코드가 미확정 서사를 코드로 안 박은 그 원칙 유지).
- **골드 소모는 뺐다**: 방 건설·레벨업의 골드 비용은 07-C. B는 "천장 안에서 방 개수를 관리하는 구조"까지.
  경제 환산을 C로 몰아 B를 얇게.
- **다음(07-D 여관)**: 인과율 엔진. 그다음 07-C에서 B의 포획 + D의 여관을 인과율 수급으로, 골드 환산·레벨업·
  스탯시딩까지 한 번에 배선(A1 opus 권고 2·3도 C에서).
