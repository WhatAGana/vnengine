|---|---|

**확립된 설계 원칙 3개(시뮬에서 도출):**

3. **새 수입원을 넣으면 반드시 기존 수입원과 같은 완화/상한을 걸어라** — 5차에서 여관을 sink로만 보고

## 1. 전투 루프 개요

```
[디펜스 페이즈]  침입 웨이브(5~60) → 방 그래프를 따라 침입자 진행 → 각 방의 몹/함정이 요격 → 처치 또는 포획 → 코어 도달 판정
      │
      ▼
[정비 페이즈 10일]  보수(내구도) + 방 구성 변경 + 몹 배치(예산제) + 여관 경영 + 던전레벨업 + 가챠
      │
      ▼
[다음 웨이브] (10일 주기 제국군 + 비정기 모험가)
```

- 디펜스 결과(방어/코어피격/포로획득)와 여관 경영 결과는 `RunState`를 갱신.
- 낮(여관 경영) / 밤(디펜스)의 이중 루프가 하나로 묶임 — 둘 다 던전레벨(인과율 해소)이라는 같은 목표를 향함.

## 2. 침입자(Attacker) 생성

> **구현 상태(07-A2, 2026-07-08):** §2.2(병종 데이터/상극)·§2.3(위협 기준선 공식)은 `Core/Sim/Combat/`에 순수함수로
> 구현됨 — 아래 각 하위섹션에 실제 타입 매핑. §2.1(웨이브 규모 5~60 생성 곡선)은 **미구현**(07-B/C 스코프) —
> 이 슬라이스는 이미 구성된 `WaveDef`(병종별 수량 리스트)를 입력으로 받아 해결만 한다.

### 2.1 규모 — 5 ~ 60명 (12명 × 5파티)
- 한 웨이브당 **최소 5 ~ 최대 60명**. (시뮬 검증: 5~30은 처리량 병목 → 성장 정체. 5~60이 병목 해소)
- 규모는 회차·던전레벨에 따라 스케일. 초반 회차는 5~15, 후반은 40~60 쪽으로 편향.

```
wave_size = clamp(5, 60,  8 + loop*3 + dungeonLevel*3 + random(-4, 6))
```

### 2.2 병종 (데이터 주도)
각 침입자는 병종을 가지며 **랜덤 지정**. `UnitClassDef`로 정의:

| 병종 (예시) | HP | ATK | DEF | 특징 |
|---|---|---|---|---|
| 탱커(방패병) | 높음 | 낮음 | 높음 | 지속딜/관통에 약함 |
| 마법사 | 낮음 | 높음 | 낮음 | 근접 급습/암살에 약함 |
| 성기사 | 중 | 중 | 중 | 신성(마기저항), 물리특화에 약함 |
| 궁수/유격 | 낮음 | 중 | 낮음 | 광역/함정에 약함 |
| 성직/힐러 | 낮음 | 낮음 | 낮음 | 아군회복, 우선처치 대상 |

- **상극은 데이터 테이블** `ClassMatchup[atk][def] = 정수%`. 예: 암살계→마법사 150(×1.5), →탱커 70(×0.7). 커널은 곱하기만.
- 권장 범위 ×0.5~×2.0 (극단 피하기 — 너무 크면 특정 카운터가 전부 결정).

> **구현(07-A2):** `UnitClassId`(string 래핑 값타입)·`UnitClassDef`(Id/DisplayName/HpPct/AtkPct/DefPct/CanBeCaptured,
> 전부 정수%)·`UnitClassCatalog.Default()`(+`UnitClassIds` 상수)가 위 5행을 그대로 데이터화: 탱커=Tank(150/60/150,
> 포획불가) · 마법사=Mage(70/150/60, 포획가능) · 성기사=Paladin(100/100/100, 포획불가) · 궁수=Archer(70/100/70,
> 포획가능) · 성직=Priest(60/60/60, 포획가능). "HP/ATK/DEF 배수(hpMul 등)" 서술은 실제로는 **정수 퍼센트**
> (`HpPct`/`AtkPct`/`DefPct`, `ThreatBase * Pct / 100`)로 구현됨.
> `ClassMatchup`은 `(atk,def)→정수%` 딕셔너리(`Multiplier(atk,def)`, 미등록 쌍은 기본 100=중립). `ClassMatchup.Default()`
> 예시: Archer→Mage 150, Archer→Tank 70, Mage→Tank 70, Paladin→Mage 130. 위 서술의 "암살계"는 1편 카탈로그에
> 없는 예시 병종 — 현재는 5병종(Tank/Mage/Paladin/Archer/Priest)만 데이터화됨.

### 2.3 침입자 능력치 산출식

```
# 위협 기준선 — 던전이 "얼마나 강해졌나"를 하나의 수로
ThreatBase = w_hero*HeroLevel
           + w_loop*(LoopCount-1)          # 회차 쌓일수록 강화 (엔드게임 난이도 곡선)
           + w_placed*AvgPlacedMonsterLevel # 동적 난이도(rubber-banding) — 가중치 작게!
           + w_dungeon*DungeonLevel
           + baseOffset
# 초기 추정: w_hero=2, w_loop=8, w_placed=1, w_dungeon=3, baseOffset=20 (전부 데이터, 튜닝대상)

# 개체 능력치 = 기준선 × 병종 프로파일 + 편차
unit.HP  = max(1, round(ThreatBase * class.hpMul)  + random(-5, 5))
unit.ATK = max(1, round(ThreatBase * class.atkMul) + random(-5, 5))
unit.DEF = max(1, round(ThreatBase * class.defMul) + random(-5, 5))
```

- `AvgPlacedMonsterLevel` 항 = 강한 방어 깔면 강한 적. **가중치 작게**(시뮬: w_placed=1). 크면 "강해질수록 손해".
- `random(-5,5)` = 02 표현식의 시드 결정론적 난수 → 세이브 재현성.

> **구현(07-A2):** `ThreatWeights`(WHero/WLoop/WPlaced/WDungeon/BaseOffset)와 `ThreatFormula.Compute`가 위
> 공식을 그대로 구현. `ThreatWeights.Default()` = wHero2/wLoop8/wPlaced1/wDungeon3/baseOffset20 — 문서 초기추정과
> 동일. **핵심 설계결정(두 가중치 테이블 분리):** `w_hero` 항은 "HeroLevel"이 아니라 `heroPower`(정수 스칼라)를
> 받으며, 이 값은 반드시 `ThreatFormula.HeroPowerOf(StatCombatWeights, HeroStats)` =
> `StatCombatWeights.Derive(hero).CombatPower`를 거쳐야 한다 — raw 8스탯 합을 직접 대입하는 것은 금지(구조적으로
> `HeroPowerOf` 래퍼가 그 경로를 강제). `ThreatWeights`(침입자가 얼마나 강한가, 5항)와 `StatCombatWeights`(주인공
> 8스탯→전투역할 파생, 밸런스 핵심 독립 테이블)는 서로 다른 파일·타입이며 참조만 한다 — 섞으면 안 됨.
> `AvgPlacedMonsterLevel`은 이 슬라이스에 배치 몹 레벨 데이터가 없어 `CombatResolver` 내부에서 항상 0으로
> 취급(07-B 이후 실배선). 개체 능력치 산출(`unit.HP/ATK/DEF`)은 `AttackerFactory.Create`가 담당(호출순서
> HP→ATK→DEF 고정, `IRandom` 경유 결정론).

## 3. 데미지 공식 (분기형)

> 검색 결론: **단일 공식은 극단에서 깨진다. 분기가 정석.** 회차마다 스탯 범위가 넓게 변하는 이 게임엔 필수.

```
if ATK >= DEF:
    dmg = ATK * 2 - DEF          # 차감식: 폭주 안 함, ATK=DEF일 때 dmg=ATK
else:
    dmg = ATK * ATK / DEF        # 제곱식: ATK<DEF에서도 0으로 안 떨어짐 (정수나눗셈 버림)

dmg = dmg * matchupMul / 100     # 상극 배수(정수%: 150=×1.5)
dmg = max(1, dmg)                # 0 데미지 금지
```

- 정수 연산만 → 02 표현식 엔진(Int 전용) 호환.
- **대안(비율 방어)**: `dmg = raw * 100 / (100 + defPercent)` (TDS식). DEF를 감쇠율로 쓰고 싶을 때. 초기엔 위 분기형 권장(직관·튜닝 쉬움).

> **구현(07-A2):** `DamageFormula.Raw(atk,def)`가 위 분기형 원시데미지를 그대로 구현(`atk>=def`면
> `atk*2-def`, 아니면 `atk*atk/def` 정수나눗셈). `DamageFormula.Apply(atk,def,matchupPct)`가 상극 배수 적용 +
> `max(1,...)`까지 구현. 전부 정수 연산(부동소수 없음).
> **문서 §3엔 없던 확장분:** `DamageFormula.Resolve(...)`가 명중(`HitRating` vs `Evasion`)·치명타(`CritRating`)
> 판정을 추가로 얹음(`HitParams` + `IRandom` 소비, 롤 순서 고정: 명중롤 → 명중시에만 치명롤). 이 슬라이스에선
> **주인공의 코어앞1칸 요격에만** 쓰이고, 방어몹→침입자 데미지는 `Apply`만 사용(명중/치명 판정 없음). 치명타
> 배수는 데이터 미도입 → 중립값 100(배수 없음) 플레이스홀더로 취급.

## 4. 방 그래프 & 배치 예산제

> **구현 상태(07-A2):** §4.4(진행·요격, 결정론)와 §4.5(처치/포획 분기)는 `RoomNode`+`CombatResolver.ResolveWave`로
> **단순화된 형태로 구현됨**. §4.1(방 그래프/상한)·§4.2(배치 예산제)·§4.3(코스트=레어도)·§4.6(웨이브 종류 구분)은
> **미구현**(07-B/C 스코프) — 아래 각 하위섹션에 표기.

### 4.1 방 구조 (가변, 던전레벨이 상한)
- 방은 **그래프 노드**. 첫 슬라이스는 선형(1→2→3→4→5코어), 자료구조는 분기·병렬 경로 확장 가능한 그래프로 열어둠.
- **방 개수 상한 = 던전레벨에 연동**: `rooms_cap = 3 + dungeonLevel*2` (lv1=5, lv2=7, lv3=9…).
- **상한과 실제 건설은 분리**: 던전레벨 = 지을 수 있는 천장, 실제 건설 = 그 안에서 자원(골드) 소모하며 하나씩. → 레벨만 올리고 방은 안 짓는 것도 전략(자원 다른 데 투자).
- 코어(마지막 노드) 피격 = 회차 실패(1회차는 서사적 자폭 회귀로 연결).

> **미구현(07-A2 기준):** `RoomNode`는 `Defenders`(배치완료 `Attacker` 목록)와 `HasTrap`만 담는 불변 스냅샷.
> 방 상한·던전레벨 연동·그래프/분기 구조는 없음 — `CombatResolver.ResolveWave`는 `IReadOnlyList<RoomNode>`를
> **선형 순회**만 한다(첫 슬라이스는 문서가 예고한 "선형" 그대로, 그래프 확장은 07-B 이후).

### 4.2 배치 예산제 (포인트 바이)
- **총 예산 = 현재 열려있는 방 개수 × 3.** (최대 방이 아니라 **현재 방** 기준 — 최대 기준이면 "방 최소, 몹 최대"가 정답이 돼 방 시스템이 죽음)
- 각 몹은 **코스트(레어도)**를 가짐. 예: 임프1, 고블린1, 오크2, 하이오크3, 서큐버스3, 데스나이트4, 고위마족5.
- 예산 내에서 자유 배치: 방3개(예산9)면 서큐(3)+데스나이트(4)+오크(2)=9, 또는 고위마족(5)+임프×4(4)=9 등.
- **두 겹 제약**(총예산 + 개체코스트)이 "질 vs 양"의 고민을 만듦. 이게 "머리 쓰는 배치"의 핵심.

> **미구현(07-A2 기준):** `RoomNode.Defenders`는 이미 배치가 끝난 `Attacker` 목록을 그대로 받아 스냅샷할 뿐,
> 예산·코스트·`MonsterDef` 개념은 없음. "배치 결과를 소비"만 하고 "배치 자체"는 07-B 스코프.

### 4.3 코스트=레어도 원칙
- **코스트 대비 효율은 대체로 평평하되, 구간마다 다른 강점**:
  - 저코스트 다수(임프·고블린): 광역 분산, 물량 견제, 소모품
  - 고코스트 소수(고위마족·데스나이트): 단일 강적(탱커·성기사) 처리, 방 봉쇄
  - 중간(오크·서큐버스): 범용 + 특수기능(서큐버스=아그네스 해금 트리거 등)
- → "9포인트 쪼개기"가 침입자 병종 구성(상극)에 따라 매번 달라짐. 예산제 + 상극이 맞물려 전략 깊이.
- **몰아넣기 vs 분산**: 몰아넣기=한 방 화력집중(강적 처리, 다른 방 빔), 분산=전 방 마모(물량 대응, 강적 놓칠 수). 방 그래프 분기 구조면 경로예측 게임으로 심화.

> **미구현(07-A2 기준):** 코스트/레어도 데이터(`MonsterDef.Cost`)와 예산 배분 로직 자체가 없음(§4.2와 동일 사유).

### 4.4 진행·요격 (결정론)
```
각 침입자: 입구 → 경로 따라 방 통과 → 방의 몹/함정이 요격(데미지공식) → HP 감소
  - HP ≤ 0 → 처치. (단 특정 병종/조건은 포획 분기)
  - 코어 도달 → 코어 피격
몹도 반격받아 HP 감소(양방향). 몹 사망 시 그 방 방어력 하락.
```
- **전투 해상도는 결정론적**(시드 고정) → 리플레이·세이브·테스트 재현.

> **구현(07-A2):** `CombatResolver.ResolveWave`가 위 흐름을 그대로 구현. 웨이브의 각 침입자를 `AttackerFactory`로
> 생성한 즉시 `rooms`를 선형 순회하며 각 방의 `Defenders`가 `DamageFormula.Apply`로 요격(HP는 로컬 변수로만
> 추적, 원본 `Attacker`/`RoomNode`는 불변 유지). 모든 방을 생존 통과하면 **코어 앞 1칸**에서 주인공이
> `DamageFormula.Resolve`로 단독 요격(명중/치명 판정 포함, §13.1의 "코어 앞 1칸 배치" 개념을 이 함수 내부
> 로직으로 반영 — 별도 배치 데이터는 아직 없음). 몹의 반격으로 인한 주인공 피해(문서의 "몹도 반격받아... 양방향")는
> **미구현**(주인공은 이 슬라이스에서 순수 요격자, HP/방어 소모 없음).
> 같은 시드+같은 배치+같은 웨이브 → 같은 `CombatResult`(rng 호출 순서 고정) — 순수함수·결정론.

### 4.5 처치 vs 포획 분기
- 처치와 별개로 **제압(포획)** 판정. 조건: 함정으로 격파(마르타), 심부유인+주인공STR(드보라), 서큐버스 보유(아그네스) 등.
- 포획 시 침입자는 죽지 않고 `RunState.Captives` + 감옥으로. → 로스터 이벤트 + 인과율(레벨재료).
- **처치 vs 포획 = 골드 vs 성장 트레이드오프** (§6.2 약탈골드 참고).

> **구현(단순화, 07-A2):** `CombatResolver`는 침입자가 처치되는 순간의 방이 `HasTrap == true`이고
> `attacker.CanBeCaptured`(병종 데이터)이면 `Captured`, 아니면 `Killed`로 분류. 문서에 예시된 히로인별 개별
> 포획조건(마르타=함정, 드보라=심부유인+STR, 아그네스=서큐버스 보유 등)은 **미구현** — 현재는 "함정 존재 여부"
> 하나로만 분기하는 최소구현.

### 4.6 웨이브 종류
- **10일 주기 제국군**: 정기. 신앙 계열(성기사·성녀·이단심문관) 편향. 서사의 "정기 배달".
- **비정기 모험가**: 랜덤. 병종 다양, 골드 드랍 편향.

> **미구현(07-A2 기준):** `WaveDef`는 `Intruders`(병종별 수량 리스트)만 가지며 `Kind`(제국군/모험가) 구분 필드가
> 없음. 웨이브 종류별 편향·생성 로직은 07-B/C 스코프.

## 5. 던전 레벨 — 게임의 중심 성장 축

던전 레벨이 여러 시스템을 동시에 밀어올림: 방 개수 상한(§4.1) + ThreatBase 강화(§2.3) + 히로인 해금(마르타 lv3) + 여관 손님풀.
→ "던전 레벨을 올린다"가 유저의 명확한 상위 목표. 성장 지표가 하나로 수렴.

### 5.1 레벨업 3경로 (인과율 = 레벨 재료)
1. **골드 소비**: 자원 전환 레버. 골드는 방건설·가챠·여관과 경쟁.
2. **포로 방면**: 플레이 방식 레버. 포획 위주 디펜스를 설계해야만 얻음(함정·포획몹 섞기). = 세계관("생명/욕망으로 인과율 해소")의 메커니즘화.
3. **여관 번영**: 손님 만족 = 인과율 배출(§7). 시뮬에서 이 3번째 축이 성장 정체를 해소함(포로만으론 부족).

### 5.2 레벨업 비용 (지수 곡선 — 상대화)
```
levelup_cost = levelup_base * (dungeonLevel ^ levelup_exp)
# 초기 추정: base=120, exp≈1.32 (시뮬: exp 1.6은 너무 가파름→1.32로 완화)
```
- 비용이 레벨 따라 오르되, 수입도 같이 오르므로 "몇 웨이브에 레벨 하나"가 대체로 일정 유지(상대화).
- 후반 소프트캡(시뮬: ~14레벨) 자연 발생. "최종 도달 레벨"을 정하고 exp로 튜닝.

### 5.3 방면 = 타락한 채 약화되어 돌아감 (세계관 정합)
- 풀어준 제국군은 인과율을 남기고 감 → 광신이 깨진 상태로 귀환 → **다음엔 약하거나, 제국 내부에 회의를 퍼뜨림**.
- "풀어주기 = 적 약화 + 레벨업"이라 손해감 없음. 성녀·마르타 서사(신앙 붕괴)와 연결. 제국 붕괴 복선.

## 6. 경제 — 골드/마석/인과율

### 6.1 통화 구조 (이중 통화)
- **골드**: 주 수입. 약탈 + 여관. 소비: 방건설·레벨업·여관투자·몹강화·마석구매.
- **마석**: 가챠 전용. 골드로 구매(환율 조절 밸브1). 가챠 비용은 마석(밸브2).
- → 두 지점에서 인플레 통제. 골드→마석→가챠로 한 단계 끼운 게 밸브를 둘로 만듦.

### 6.2 약탈 골드 산식 (제곱근 완화)
```
loot_gold(개체) = gold_base + isqrt(ThreatBase) * gold_threat_k  (+ 병종가중치, ±편차)
# 초기 추정: gold_base=5, gold_threat_k=3
# 정수 제곱근(isqrt): 엔진 정수전용이므로 근사(뉴턴법/룩업)

처치: 골드 100%
포획: 골드 50% + 인과율(레벨재료)   ← 처치=돈, 포획=성장 트레이드오프
```
- 제곱근이 피드백 기울기를 꺾음: ThreatBase 100→400(4배)이어도 보상 2배만. (시뮬 검증: 이게 인플레 차단 핵심)
- 병종 가중치: 고위직(성기사단장·이단심문관) 배수↑, 잡졸↓.

### 6.3 가챠 비용 (회차 리셋)
```
gacha_cost = gacha_base + pulls_this_loop / 3    # 뽑을수록 비싸짐
```
- **pulls_this_loop 카운터는 RunState(런)에 둠 → 회차마다 리셋.** (메타에 두면 엔드게임에서 가챠가 막힘 — 반드시 런)
- 매 회차 초반 싸게 → 후반 비싸게 리듬 반복. 무한 가챠 억제 + 마석 소비 sink.

### 6.4 골드 sink 우선순위 (시뮬 확립)
지출은 이 순서로 (성장을 sink보다 우선해야 성장이 안 죽음):
```
1) 인테리어 내구도 수리   (수입 게이트라 최우선 — §7.2)
2) 던전 레벨업           (성장)
3) 여관 투자(직원/메뉴)   (후반 잉여 흡수 + 수입 루프)
4) 몹 강화               (계승몹 파워업, 회차당 상한)
5) 잉여 골드 → 마석 → 가챠
```
- **성장 지출(레벨업)이 마석/가챠보다 먼저.** 시뮬 1차에서 sink(마석 70%)를 성장보다 먼저 처리해 성장이 죽었음 → 순서 교정.

### 6.5 인플레 방지 종합 (5원칙)
1. **피드백 차단**: ThreatBase가 보상·난이도 양쪽에 있으나, 제곱근·상대화로 폭주 안 함.
2. **제곱근 완화**: 약탈 골드가 적 강함에 선형(×) 아닌 sqrt로 붙음.
3. **비용 상대화**: 레벨업 비용·상시지출이 수입과 같은 곡선으로 상승 → 비율 고정.
4. **상시 sink**: 여관 직원급여·감옥운영비·내구도 수리가 후반까지 골드 흡수.
5. **하드캡**: 계승몹(08)·레벨 소프트캡 → 절대 무적 방지.
- **상시 지출 총합 < 평균 수입의 절반**을 기준선(시뮬: 평균 28%, 정체기 46%).

## 7. 여관 타이쿤 레이어 (= 인과율 엔진)

> 시뮬 5→6차 핵심 교훈: 여관을 sink로만 보면 안 됨. **여관은 벌기도 하므로**, 골드 수입에 완화+상한 필수.
> 여관의 성격을 "골드벌이"가 아니라 **"인과율 엔진(레벨업 주력 축)"**으로 정의 → 돈은 전투, 성장은 여관 역할분리.

### 7.1 여관 3요소
| 요소 | 성격 | 효과 | 비용 |
|---|---|---|---|
| **직원** | 초기고용 + 상시급여 | 수용력↑ → 손님↑ → 인과율↑ | 고용비 + 매웨이브 급여(상시 sink) |
| **인테리어 내구도** | 강제 상시 | 0이면 손님 안받음(수입 게이트) | 자연감소 → 수리비 지속 |
| **메뉴 개발** | 일회성 대형투자 | 손님단가/인과율 영구↑ | 큰 골드(후반 잉여 흡수처) |

### 7.2 여관 수입 산식 (완화 + 상한 — 5차 폭주 교훈)
```
if decor > 0:
    guests = min(staff*2 + menu_lv, 25)      # 손님수: 여관규모에만 연동(던전레벨 무관), 상한 25
    inn_gold  = min( isqrt(guests)*8 + menu_lv*3, 300 )   # 골드: 제곱근+상한(여관은 돈벌이 주력 아님)
    inn_karma = guests                        # 인과율: 손님수 그대로(레벨엔진 주력)
    decor -= 2                                # 내구도 자연감소
```
- **손님수를 던전레벨과 분리** → 던전레벨 오를 때 여관수입 자동폭증 차단(5차 폭주 원인).
- **여관 골드비중 목표 <10%** (시뮬: 7%). 여관은 인과율로, 돈은 전투로.

### 7.3 세계관 정합
- 여관 = 코어 인과율 배출구(초대 주인 설치). 손님이 많고 만족할수록 인과율 배출↑ = 던전 안정 = 레벨업.
- 디펜스(포로 방면)와 여관(손님 만족)이 **같은 목표(인과율 해소)의 두 경로** → 낮/밤 이중 루프가 하나로.

```
      - 여관수입 상한 작동 (던전레벨 올려도 여관골드 폭증 안함)
      - 내구도 0이면 여관수입 0
      - 지출 우선순위 (레벨업이 가챠보다 먼저)
```

### 설계 불변식 (테스트로 강제)
1. 모든 Resolve/Rule은 입력 상태를 변형하지 않는다(불변).
2. 같은 시드+같은 배치 → 같은 결과(결정론).
3. 데미지 ≥ 1 (0 금지).
4. 테마 문자열/매직넘버 하드코딩 없음(전부 데이터).
5. 가챠 pull 카운터는 런에 귀속(회차 리셋).
6. 여관 골드수입은 상한을 넘지 않는다(인플레 차단).

## 11. 하루 / 페이즈 구조 (시간 단위)

90일 = 9주기(각 10일). 한 주기의 구조:

```
1~9일차: 정비 페이즈 (방구성 변경·몹배치·여관경영·던전레벨업·주인공강화·가챠)
         └ 이 기간에 모험가 웨이브가 확률적 난입 가능 (기본 확률 ~40%/주기, 데이터)
10일차:  제국군 웨이브 확정 (10일 주기 정기 파병)
웨이브 오는 날 = 정비 불가.
9일차에 상태 저장(세이브) 확인을 띄운다.
```

- **모험가 웨이브**: 정기 아님. 1~9일차 확률 난입. **사전 예보 없음** — 단, 여관의 "주변 마을 파견(재료수급)" 중
  **소문 수집**으로 미리 감지 가능(§12.3). 파견 자주 → 대비력↑, 안 하면 기습.
- **경제 영향(밸런싱 주의)**: 시뮬은 제국군9회만 셈. 실제는 제국군9 + 모험가난입α라 **웨이브 총량이 더 많음** →
  회차당 수입·전투빈도가 시뮬보다 높음. 모험가 난입 확률이 회차당 총수입을 좌우하므로 튜닝 시 핵심 변수.

### 13.1 배치 제약
- 주인공은 **배치 예산(코스트) 제외**로 배치. 대신 **코어룸으로부터 1칸 앞의 방까지만** 배치 가능.
- → 주인공 = **최종 방어선**. 앞방은 몹으로, 뚫고 들어온 강적을 코어 앞에서 주인공이 처리.
- **서사 정합**: 드보라 씬4(심부 유인 + 주인공 직접 제압)와 자동 일치 — 드보라는 코어 근처까지 뚫어야 주인공과 만남.

### 13.2 8스탯 (STR/INT/DEX/AGI/HP/MP/DEF/LUK)
1편부터 **다스탯 전부 노출**(시뮬 10차: 밸런스 성립). 각 스탯 상한 **999**.

| 스탯 | 전투 역할 |
|---|---|
| **STR** | 물리 공격력 (물리 빌드 주력) |
| **INT** | 마법 공격력 + MP 효율 (마법 빌드 주력) |
| **DEX** | 명중 + 치명타 확률 |
| **AGI** | 행동 속도 + 회피 |
| **HP** | 체력 |
| **MP** | 스킬 자원 |
| **DEF** | 방어력 |
| **LUK** | 치명·포획률·드랍·가챠 보정 |

- **데이터 주도(확장 대비 필수)**: 스탯을 하드코딩하지 말고 `StatDef` 테이블로 정의. 전투공식은 스탯을 **간접 참조**
  ("물리공격 기여 스탯" 식). → 확장판에서 스탯 관여도를 데이터로 늘려 하드코어(3번)로 확장. **2편 "순수 힘 던전"**
  (차원 연결 없이 주인공 힘만으로 코어 생성 = 8스탯이 코어 동력)의 골격이 여기서 놓임. 세이브에 8스탯 전 필드 확보.

### 13.3 스탯 성장 = 인과율 소비 (§14와 연결)
- 주인공 스탯은 **인과율 포인트**로 올림(골드 아님). → 감옥의 존재 이유(포로=인과율=주인공 강화).
- **구간지수 비용**(시뮬 10차): 스탯 낮을 땐 싸고, 높을수록 급증. → 999는 "극후반 로망"(40회차에도 미도달),
  전스탯 만렙 불가 → **빌드 선택 강제**(물리/마법/균형/럭포획 등 차이 770).
  ```
  stat_cost(cur): cur<100→1, <250→2, <450→3, <650→5, <800→9, <950→16, else→28  (초기추정, 튜닝대상)
  ```

### 13.4 소문 수집 (여관 파견 기믹)
- 여관의 "주변 마을 파견(재료수급)" 중 확률적으로 **모험가 웨이브 소문** 획득 → 사전 대비.
- **정보 정밀도**: 규모나 시기 중 **하나만** 알려주는 정도(둘 다면 랜덤긴장 죽고, 없으면 무용). → 07-D에서 확정.

## 14. 인과율 = 통합 자원 (시뮬 7·8차)

인과율이 골드와 별개의 **제2 통화**. 수급과 소비가 분리된 통합 자원.

### 15.1 수급 / 소비
```
수급처: 여관(손님 만족, 손님3당 인과1) + 감옥(포로 방면)
소비처: 던전 레벨업(재료) + 주인공 8스탯 강화
```
- 유저는 인과율을 **던전 vs 주인공**에 배분 → 두 번째 전략 축(골드는 방/가챠/여관 배분, 인과율은 던전/주인공).

### 15.2 공급과잉 방지 (7차 교훈 → 8차 해결)
- **7차 발견**: 수요에 상한(회차당) 있고 공급엔 없어 인과율 무한적립(저금 960 = 인플레 재발).
- **원칙 재확인**: 새 자원에도 "수요에 상한/완화" 필요(골드에서 배운 것과 동일).
- **8차 해결**: 주인공 강화 회차상한 제거 + **구간지수 비용**으로 남는 인과율을 "더 키우는 데" 흡수.
  8스탯(10차)이 갈 곳을 늘려 흡수력↑ → 저금 폭주 대부분 해소 + 배분이 유한자원 선택이 됨.
- **밸런싱 목표**: **공급 ≈ 수요**로 맞추기. 남으면 배분 선택이 무의미(다 채워짐), 모자라면 성장 정체.
  정밀값은 실측 튜닝(시뮬 `sim_karma2.py`, `sim_8stat.py`로 사전 확인 가능).

| 인과율 저금(bank) 잔액, 주인공 8스탯 | **메타** | 회차 넘어 유지(주인공 성장은 영속) |
