# 08 — 계승 · 스킬증여 · 육성 (설계 사양)

> **아직 구현 안 됨.** 06(루프 상태)·07(전투/경제) 위에 얹는 슬라이스. 원칙 유지: **순수 코어 · 불변 · 데이터 주도.**
> 「둥지짓는 드래곤」식 n회차 기믹: 몹을 다음 회차로 계승 + 스킬증여로 빌드 조합 + 육성 반복.
> **핵심 과제: 리텐션(성장 재미)은 살리되 무한강화는 차단.** 계승은 회차를 넘는 유일한 강화 벡터라 방치하면 인플레 폭탄.
> 시뮬(`sim_inherit.py`)로 5축 상한이 무한강화를 차단함을 검증(파워 30→200에서 정지, 스킬슬롯 3 고정).

관련: [06 회차 루프](06-loop-and-state.md) · [07 전투/경제](07-defense-combat.md)

---

## 0. 왜 위험하고, 왜 막기만 해도 안 되는가

- **위험**: 06에서 회귀 시 런은 리셋되지만 계승몹은 메타를 타고 넘어옴. 즉 **회차를 넘어 영구히 강해지는 거의 유일한 벡터**.
  여기에 스킬증여로 "좋은 스킬만 모으기"까지 되면 완벽한 몹이 만들어져 디펜스가 무의미해짐(07 인플레 방지 전부 무력화).
- **막기만 해도 안 됨**: 계승+육성의 "점점 강해지는 맛"이 유저가 n회차를 반복하는 이유. 너무 조이면 반복 동기 소멸.
- **목표**: 무한 성장 → **유한 성장 + 천장 + 성장이 새 도전을 부름**. 계승의 목표를 "절대 강해지기"가 아니라
  **"더 높은 회차/난이도를 감당하기"**로 프레이밍(이 프레이밍이 서면 모든 수치가 따라옴).

## 1. 무한강화 차단 5축 (시뮬 검증)

| 축 | 내용 | 차단하는 것 |
|---|---|---|
| **1. 계승 인원 소수정예** | 기본 1마리, 특정성취로 소폭확장, **하드캡 3** | 군단화 |
| **2. 스킬 슬롯 고정** | 몹당 스킬 **2~3개 고정**. 넘으면 교체 | 스킬 무한축적 |
| **3. 증여 손실** | 스킬 추출 시 **원본 소멸 + 골드비용** | 무한 최적화 |
| **4. 능력치 하드캡** | 레벨 천장 + 지수 요구치 | 절대 무적 |
| **5. 난이도 추적** | ThreatBase가 계승 강함 반영(07 §2.3) | 성장의 절대화 |

시뮬 결과: 파워 87→200에서 정확히 정지(11회차 수렴), 스킬슬롯 3 고정, 파워배율 6.7x(유한).

---

## 2. 계승 (축1)

### 2.1 계승 슬롯
- **회차 종료 시, 정해진 슬롯 수만큼 몹을 메타로 이관**(`MetaState.InheritedMonsters`).
- 기본 슬롯 1. **하드캡 3.** 확장은 무료가 아님(아래).
- 나머지 소환몹(`RunState.Summoned`)은 회귀 시 전부 폐기(06 Regress).

### 2.2 슬롯 확장 조건 (명확히 — 시뮬에서 조건 모호하면 확장 안 됨)
- 슬롯 확장은 **특정 성취**로만. 예: "진엔딩(5회차) 도달", "특정 히로인 전원 해금", "던전레벨 N 도달".
- **회차 숫자로 자동 확장 금지**(그럼 반복이 숙제가 됨). 성취 기반이라 열심히 판 유저가 도달.
- 확장해도 **하드캡 3 불변.**

### 2.3 계승 대상 선택
- 회차 종료(회귀 직전) UI에서 유저가 계승할 몹을 슬롯 수만큼 선택 → `RegressionInput.MonstersToInherit`(06 §3).
- 선택 안 한 몹은 소멸.

---

## 3. 스킬 시스템 (축2)

### 3.1 스킬 슬롯 고정
- 몹당 스킬 슬롯 **2~3개**(데이터로 몹 종별 차등 가능: 고코스트 몹은 3, 저코스트는 2 등).
- 슬롯이 꽉 찬 상태로 새 스킬을 넣으면 **기존 것 하나를 골라 교체**(버림). → 무한축적이 "선택과 교체"로.

### 3.2 스킬 예시 (데이터 주도)
「둥지짓는 드래곤」식 특징 스킬: 초회복, 영웅베기(특정 병종 특효), 광역, 관통(방어무시), 흡혈, 보호막 등.
- 각 스킬은 `SkillDef`로 정의(효과·발동조건·수치). 커널은 데이터를 읽어 적용.

### 3.3 스킬 상성/조합 (완벽 만능몹 방지)
- 특정 스킬끼리 **시너지**(초회복+보호막=탱커빌드) 또는 **상호 페널티**(공격특화끼리는 감소).
- → "모든 좋은 스킬을 한 몹에" 대신 "이 몹은 탱커빌드, 저 몹은 딜러빌드"로 분화. 만능몹이 안 나옴.

---

## 4. 스킬증여 (축3 — 무한 최적화 억제)

### 4.1 증여 = 추출 + 이식, 손실 동반
```
transfer_skill(src, dst):
    - src(원본 몹)에서 스킬 하나 추출
    - dst에 이식 (슬롯 꽉차면 교체)
    - src는 소멸 (또는 크게 약화)   ← 핵심: 스킬 뽑으면 원본 희생
    - 골드 비용 소모               ← 07 골드 sink와 연결
```

### 4.2 왜 원본 소멸인가
- 무손실이면 유저가 완벽조합 나올 때까지 무한 갈아끼워 최적해로 수렴 → 빌드 고민 사라짐.
- 원본을 희생해야 하면 "능력 좋은데 스킬 별로인 A"에게 "스킬 좋은데 능력 별로인 B"의 스킬을 주려면 **B를 버려야** 함.
  → "정말 이 조합을 완성할 가치가 있나"를 고민. 자원(몹) 소모로 무한 최적화 억제.

### 4.3 (선택) 이식 성공률
- 실패 시 스킬 손실 등 리스크. 호불호 갈림 → 초기엔 미적용, 확정적 손실(원본소멸+비용)만으로 충분.

---

## 5. 능력치 육성 & 하드캡 (축4)

### 5.1 레벨업 (골드 소모 = 07 sink)
- 계승몹은 골드로 레벨업/강화(07 §6.4 지출 우선순위 4번 "몹 강화").
- **회차당 강화 상한**(시뮬: mob_up ≤ 24) → 한 회차에 급성장 방지, 여러 회차 걸쳐 성장.

### 5.2 하드캡
- **능력치 하드캡**(시뮬: STAT_CAP=200). 이 이상 안 오름.
- 레벨업 요구 경험치를 **지수 곡선**으로 → 후반엔 사실상 정체(경험치 인플레 방지, 07 §5.2와 동일 원리).
- → 완성된 계승몹도 "무적"이 아니라 "강한 유닛 하나". 시뮬: 파워 200 수렴 후 평평.

---

## 6. 난이도 추적 (축5 — 자동 균형)

- 07 §2.3 ThreatBase의 `AvgPlacedMonsterLevel` 항에 **계승몹 스탯/스킬 수준 반영**.
- → 강한 계승몹 배치할수록 침입자도 강해짐. 계승 이득이 "절대 무적"이 아니라 "높은 난이도 감당 수단"으로 상대화.
- 07 골드 인플레를 "비용을 수입 곡선에 묶어" 잡은 것과 동일 원리(절대값 아닌 비율 고정).
- **가장 우아한 축**: 다른 4축이 "상한을 씌운다"면 이 축은 "성장을 도전과 물린다". 무한성장이 무한난이도와 균형.

## 7. 상태 귀속 (06 런/메타)

| 데이터 | 층 | 이유 |
|---|---|---|
| `InheritedMonsters[]` (계승몹, 스킬·스탯 포함) | **메타** | 회차 넘어 유지 — 계승의 본질 |
| 계승 슬롯 수, 확장 성취 플래그 | **메타** | 영속 |
| 이번 회차 소환몹(`Summoned`) | **런** | 회귀 시 폐기(계승 선택분만 메타로) |
| 스킬증여 진행중 상태 | **런** | 정비 페이즈 내 처리 |

## 8. 코어 모델 (구현 대상)

```csharp
// Core/Sim/Monster/**
SkillDef        { Id, DisplayName, Effect, Trigger, Values[], Synergies[], Penalties[] }  // 데이터
MonsterInstance { DefId, Level, Stat, Skills[](≤SkillSlots), ... }                          // 개체
InheritanceRule {
    int SlotCap = 3;                          // 축1 하드캡
    int SkillSlots(MonsterDef);               // 축2 (몹별 차등가능)
    int StatCap = 200;                        // 축4 하드캡
    CanExpandSlot(MetaState) → bool;          // 축2 성취기반 확장
}
SkillTransfer   { Transfer(src, dst, gold) → (newDst, destroyedSrc, goldSpent) }            // 축3 순수함수
MonsterGrowth   { LevelUp(mob, gold) → newMob;  int PerLoopCap = 24; }                       // 축4 회차상한
// 축5는 07 ThreatFormula가 계승몹 반영(신규 코드 아님, 데이터 연결)
```

- 모든 Transfer/Growth는 **순수 함수**(입력 불변, 새 인스턴스 반환).
- 스킬 효과·시너지·수치는 전부 데이터. 하드캡 값(3/2~3/200/24)도 데이터(튜닝대상).

## 9. 구현 체크리스트 (클로드 코드용)

```
[ ] Core/Sim/Monster/SkillDef.cs        — 스킬 정의(효과·시너지·페널티, 데이터)
[ ] Core/Sim/Monster/MonsterInstance.cs — 개체(스킬슬롯 제한)
[ ] Core/Sim/Monster/InheritanceRule.cs — 5축 상한(슬롯캡/스킬슬롯/스탯캡/확장조건)
[ ] Core/Sim/Monster/SkillTransfer.cs   — 증여 순수함수(원본소멸+비용)
[ ] Core/Sim/Monster/MonsterGrowth.cs   — 레벨업(회차당 상한, 스탯 하드캡)
[ ] 06 MetaState.InheritedMonsters 연결 + 07 ThreatFormula에 계승몹 반영(축5)
[ ] 데이터: 스킬SO(초회복/영웅베기/광역/관통/흡혈/보호막…), 시너지테이블, 하드캡값
[ ] Unity/Sim/Monster/** — 계승선택UI, 스킬증여UI, 육성UI
[ ] EditMode 테스트:
      - 계승 인원 하드캡 3 초과 불가
      - 스킬 슬롯 초과 시 교체(축적 안됨)
      - 증여 후 원본 소멸 + 골드 차감
      - 능력치 하드캡 200 초과 불가
      - 회차당 강화 상한 24
      - 20회차 반복 후 최강몹 파워가 상한 수렴(무한강화 없음) ← sim_inherit.py 재현
      - 슬롯 확장이 성취기반으로만(회차숫자 자동확장 없음)
      - 시너지/페널티 정확 적용
```

### 설계 불변식
1. 계승 인원 ≤ 하드캡(3), 항상.
2. 몹 스킬 수 ≤ 스킬슬롯, 항상(초과 시 교체).
3. 증여는 원본을 소멸(또는 약화)시킨다 — 무손실 증여 금지.
4. 능력치 ≤ 스탯 하드캡, 항상.
5. 계승몹 강함이 ThreatBase에 반영된다(축5).
6. 모든 Transfer/Growth 순수함수(입력 불변).

## 10. 밸런싱 노트

- 하드캡 값(슬롯3/스킬슬롯2~3/스탯200/회차강화24)은 초기 추정. 플레이테스트로 튜닝.
- **핵심 튜닝 질문**: "완성된 계승몹 1마리가 그 시점 침입 웨이브를 얼마나 감당하나?" — 100% 무쌍이면 하드캡↓ 또는 축5 강화.
  적당히 강하되 나머지 배치가 여전히 필요한 수준이 목표.
- 스킬 시너지/페널티는 "만능 조합"이 안 나오게. 특정 3스킬 조합이 압도적이면 페널티 추가.
- 계승 프레이밍 재확인: 수치가 "절대 강함"으로 흐르면 축5(난이도 추적) 가중치를 올려 "난이도 감당 수단"으로 되돌림.
- 시뮬 `sim_inherit.py`로 새 하드캡 값 넣어 무한강화 차단 재확인 가능(유니티 이식 전).
